Ten consecutive earlier log entries had asserted that plain bare URLs were standing practice and "confirmed working" for the live-link notification; the real message history showed almost every run actually sent Slack's `<url|label>` bracket syntax instead, and on 12 July the third link's label was long enough that Slack silently truncated it with an ellipsis, producing exactly the broken-looking link the client kept reporting. This run tested genuinely bare URLs first, but reading the sent message back showed Slack itself re-wraps any bare URL into `<url|label>` for display using the URL text as the label, which sat at 57-60 characters, close to the ~55-65 char range that caused the 12 July truncation. The verified, truncation-proof fix, sent as a same-run correction and confirmed clean on read-back: plain markdown `[short title](url)`, using a short human-readable title (not the URL text) as the label, one per line with a blank line between entries. A short title label can never hit the truncation width regardless of how long the slug is. **Use `[title](url)` markdown for every future Slack notification. Do not send bare URLs (Slack re-wraps them with the URL as the label) and do not hand-write `<url|label>` (same truncation risk on long slugs).**

## Mistakes to avoid
- Do not push to a feature branch; deploy only fires on `main`.
- Do not modify index.html / styles.css / script.js during a content run (sitewide changes are a separate job).
- Do not pick work by date; progression is by pointer (`next_index`).
- Do not enable tracking IDs or wire Netlify without an explicit instruction.
